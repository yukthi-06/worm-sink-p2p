# Walkthrough - Fix Sender Watchdog and Receiver Heartbeat Reply

We have implemented a dual-sided heartbeat protocol mechanism to resolve the hang/deadlock issue when the receiver is killed and restarted.

## Changes Made

### `TransferEngine.java`
- **Receiver Heartbeat Reply**: In `handleProtocolFrame()`, modified the `HEARTBEAT` case so that when a receiver receives a heartbeat, it immediately replies with a heartbeat frame back to the sender.
- **Message Timestamp Updates**: Updated `onMessageReceived()` to record `lastHeartbeatReceivedMs = System.currentTimeMillis();` for any incoming message, ensuring that active transfers or heartbeat responses reset the watchdog.
- **Sender Heartbeat Watchdog**: Implemented `startSenderHeartbeatWatch()`, which schedules a watchdog check every 2 seconds. If no messages or heartbeats are received from the receiver for 8 seconds, it assumes the receiver died. It sets `receiverDisconnected = true`, asynchronously resets the signaling session, and counts down the `sessionLatch` to start a new peer connection.

## Verification
- Once you compile and run the project, the sender will now actively detect receiver disconnection within 8 seconds of silent crash or process termination. It will automatically reset the signaling server state and start listening for a new peer connection/offer.
- A newly started receiver will then be able to connect to the fresh offer instead of grabbing a stale one.
