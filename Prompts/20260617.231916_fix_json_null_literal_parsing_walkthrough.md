## Root Cause

The signaling server returns `{"offer":null,"answer":null,...}` for a new session.
`SimpleJson.getField()` matched the JSON token `null` using the unquoted-value regex
group `([^,}\s]+)`, returning the Java **String** `"null"` instead of Java `null`.

Since `"null" != null` in Java, `pollForOffer` and `pollForAnswer` both immediately
triggered on their very first scheduler iteration (0-delay), passing the bogus string
`"null"` as an SDP to `peerConnection.setRemoteDescription()`. The native WebRTC engine
hung indefinitely on this invalid SDP — explaining why both sender and receiver appeared
permanently stuck with no error message.

## Fix Applied

### [SimpleJson.java](file:///x:/Projects_X/0_Active/1_Java_Active/AI_Assisted/WormSink_Java_GITHUB/wormsink-signaling/src/main/java/org/wormsink/signaling/SimpleJson.java)

Added a guard in `getField()` to return Java `null` when the matched token is the literal
string `"null"`:

```java
// JSON null literal → Java null
if ("null".equals(val)) {
    return null;
}
```

This single 3-line change fixes the hang across all code paths that use `SimpleJson.getField`
including offer, answer, fileHash, fileName, fileSize, etc.
