# User Prompt

Build a production-quality Java 21 command-line application named **WormSink** that functions similarly to Magic Wormhole, but optimized for large-file peer-to-peer transfer over the Internet using WebRTC data channels.

## High-Level Goals

Create a secure, resumable, cross-platform file transfer tool that:

* Transfers files directly between peers using WebRTC DataChannels.
* Uses a lightweight signaling service running on Cloudflare Workers.
* Supports NAT traversal via STUN and optional TURN servers.
* Handles very large files (100GB+).
* Supports interruption recovery and resumable downloads.
* Has a clean modular architecture so the transfer engine can later be embedded into desktop, web, Android, or JavaFX applications.
* Uses Gradle as the build system.
* Runs entirely from the command line initially.

The design must prioritize reliability, resumability, modularity, performance, and future UI integration.

# Signaling Layer

Implement a lightweight signaling protocol.

Use Cloudflare Workers only for:

* Session creation.
* Session lookup.
* Offer exchange.
* Answer exchange.
* ICE candidate exchange.

Cloudflare Worker must never relay file data.

All file data must flow directly peer-to-peer.

Design worker API:

```http
POST /session/create
POST /session/offer
POST /session/answer
POST /session/candidate
GET  /session/{code}
```

Provide complete Worker implementation.

# Transfer Protocol

Design a custom transfer protocol above WebRTC.

Message types:

```text
HELLO
METADATA
REQUEST_CHUNKS
CHUNK
ACK
COMPLETE
```

# Chunking System

Split files into chunks.

Requirements:

* Default chunk size = 4MB.
* Configurable.
* Sequential disk reads.
* Random-access writes on receiver.

Each chunk contains:

```text
chunkIndex
offset
size
sha256
payload
```

# Integrity Verification

Calculate:

## Per Chunk

SHA-256 hash.

## Whole File

SHA-256 hash.

Receiver verifies:

* Every chunk.
* Final file hash.

Corrupt chunks must be re-requested.

# Resumable Downloads

This is a critical feature.

Requirements:

### Receiver Side

Maintain:

```json
transfer.state
```

containing:

* transfer id
* file hash
* completed chunks
* missing chunks
* destination path

Persist after every successful chunk.

### Sender Side

Keep metadata available throughout transfer.

---

# Resume Flow

If download stops:

```bash
wormsink receive apple-river-candle-mountain
```

Program should:

1. Detect existing state file.
2. Verify matching metadata.
3. Determine missing chunks.
4. Request only missing chunks.
5. Continue transfer.

Already downloaded chunks must never be re-downloaded.

# Encryption & Security

WebRTC encryption already provides DTLS/SRTP.

Additionally:

* Verify peer session code.
* Validate all metadata.
* Validate chunk hashes.
* Prevent malformed packet attacks.

Never execute received content.
