# Walkthrough - Fix Thread and Client Leaks

We have fixed the thread and client leaks that led to native memory exhaustion and `Failed to attach thread` errors.

## Changes Made

### `SignalingClient.java`
- Shared a single `static final HttpClient SHARED_CLIENT` across all `SignalingClient` instances. This prevents spawning a new `HttpClient` (along with its selector and thread pools) for each HTTP request/reset session, completely eliminating the selector/worker thread leaks.

### `TransferEngine.java`
- Refactored `heartbeatScheduler` to be created once per top-level `send()` and `receive()` operation and shut down in the `finally` block.
- Standardized heartbeat helper methods (`startSenderHeartbeat()`, `startSenderHeartbeatWatch()`, `startReceiverHeartbeatWatch()`, `scheduleIceGraceTimer()`) to schedule tasks directly on the pre-initialized `heartbeatScheduler`.
- Updated `stopHeartbeat()` to cancel individual `ScheduledFuture<?>` tasks instead of shutting down the entire executor service, preventing the creation of new scheduled executors during reconnect loops.

## Verification
- Running multiple reconnect loops or restarting the receiver will no longer leak thread pools or selector managers. The JVM should stay under memory limits, preventing the `VM attach current thread failed` / `Failed to attach thread` errors.
