# Walkthrough - Fix Sender DISCONNECTED Detection

We have modified `onConnectionStateChange` to immediately trigger the signaling session reset and peer connection reconnect loop when the sender detects the `DISCONNECTED` state.

## Changes Made

### `TransferEngine.java`
- Modified `onConnectionStateChange()` to check for `isSenderDisconnected` (when `isSender` is true and `state` contains `"DISCONNECTED"`).
- Grouped `isSenderDisconnected` with the existing `isFailed` and `isClosed` conditions. This triggers an immediate asynchronous `resetSession()` call and counts down the `sessionLatch` to start a new peer connection.

## Verification
- Once you compile and run the project, if the receiver is killed, the sender's WebRTC stack will transition to `DISCONNECTED` almost instantly. 
- The sender will immediately clear the stale offer/state from the signaling server and spin up a new connection manager with a new offer.
- The restarted receiver will poll and successfully get the fresh offer, proceeding with the transfer instead of getting stuck.
