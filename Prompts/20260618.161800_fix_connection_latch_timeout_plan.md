> 'Download failed: Connection timed out after 60s ? sender may not be ready yet, please retry
> VM attach current thread failed
> Failed to attach thread 16232'

---

# Goal Description

Fix the timeout and crash when downloading files taking longer than 60 seconds. Currently, the receiver blocks on `sessionLatch.await(60, TimeUnit.SECONDS)`. This timeout was meant for connection establishment but incorrectly limits the *entire transfer duration*. When the timeout is hit during a download, the connection is closed abruptly, causing native WebRTC callback threads to fail to attach to the VM.

We will introduce a separate `connectionLatch` to enforce the 60-second timeout only for connection establishment, while letting the transfer run indefinitely on `sessionLatch.await()` once connected.

## Proposed Changes

### [TransferEngine]

#### [MODIFY] [TransferEngine.java](file:///x:/Projects_X/0_Active/1_Java_Active/AI_Assisted/WormSink_Java_GITHUB/wormsink-transfer/src/main/java/org/wormsink/transfer/TransferEngine.java)

- Add a `connectionLatch` field:
  ```java
  private volatile CountDownLatch connectionLatch = new CountDownLatch(1);
  ```
- Implement `releaseLatches()` helper to count down both `sessionLatch` and `connectionLatch`:
  ```java
  private void releaseLatches() {
      sessionLatch.countDown();
      if (connectionLatch != null) {
          connectionLatch.countDown();
      }
  }
  ```
- Replace all existing instances of `sessionLatch.countDown()` with `releaseLatches()`.
- In `onDataChannelStateChange()`, when the data channel reaches `OPEN` on the receiver, count down `connectionLatch` only (do not call `releaseLatches()`).
- In `receive()`, initialize `connectionLatch = new CountDownLatch(1)` and wait on it first:
  ```java
  boolean released = connectionLatch.await(CONNECT_TIMEOUT_MS, TimeUnit.MILLISECONDS);
  ```
- If connection succeeds, wait indefinitely on the `sessionLatch`:
  ```java
  sessionLatch.await();
  ```
