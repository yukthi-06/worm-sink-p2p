## Prompt

file downloaded successfully with the correct file name, however the logs are showing some error.

Sender log ends with:
  Transfer completed successfully!
  VM attach current thread failed
  Failed to attach thread 22304

Receiver log ends with:
  Download completed successfully!
  VM attach current thread failed
  Failed to attach thread 16912

---

## Root Cause

"VM attach current thread failed" and "Failed to attach thread NNNNN" are printed by the
native `webrtc-java` JNI layer. After `peerConnection.close()` returns on the Java side,
the native C++ WebRTC threads (ICE agent, DTLS transport, SCTP/DataChannel) still have
callbacks queued. They try to call back into the JVM to deliver those callbacks, but the
JVM has already begun shutting down → JNI `AttachCurrentThread()` fails → the native code
prints the warning to stderr.

The transfer itself is 100% correct. This is purely cosmetic stderr noise at exit.

## Fix — PeerConnectionManager.java

Insert a 300 ms `Thread.sleep()` in `close()` between `peerConnection.close()` and
`factory.dispose()`. This gives the native threads time to flush their final callbacks
while the JVM is still fully alive, so `AttachCurrentThread()` succeeds and the callbacks
are delivered silently (and then ignored, since `isClosed = true`).

```java
try { Thread.sleep(300); } catch (InterruptedException ignored) {}
```
