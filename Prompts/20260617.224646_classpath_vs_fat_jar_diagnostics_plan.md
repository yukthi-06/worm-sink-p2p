## Goal
Bypass the JNI `VM attach current thread failed` error and connection hang by switching from fat jar execution to standard classpath execution using `run-gradle-based.bat`.

## Context & Root Cause Analysis
- The flat fat jar extracts and packages dependency class files into a single jar file.
- Under some JVM configurations, when WebRTC's C++ threads call back into Java via JNI, JNI attempts to attach the native thread to the JVM. If classes are loaded in a specific manner or if there are package/module accessibility constraints under `-jar` mode, JNI thread attachment can fail (`VM attach current thread failed`).
- Running with standard classpaths (where all dependency jars remain separate and are loaded normally via the classpath loader) avoids classloader lookup mismatches and JNI environment lookup issues.

## Proposed Actions
1. Instruct the user to run the application via `run-gradle-based.bat` (which builds the standard distribution using Gradle's `installDist` task and runs it with all libraries in a standard classpath directory structure).
2. Perform the sender and receiver commands using `run-gradle-based.bat`.
