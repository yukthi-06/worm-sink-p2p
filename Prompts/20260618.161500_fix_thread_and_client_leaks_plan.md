# User Prompt

> on restarting receiver, it is able to download fine.
> but the issue is the receiver is getting disconnected after some time with error 'Failed to attach thread'.
> Once manually restared, it again starts downloading.

---

# Goal Description

Fix the thread leaks and native memory exhaustion that lead to `VM attach current thread failed` / `Failed to attach thread` errors during transfer reconnects/retries.

We identified two major leaks:
1. **HttpClient Leak**: `SignalingClient` creates a brand new `HttpClient` in its constructor. Since `SignalingClient` is instantiated repeatedly (on every peer connection attempt and on every async session reset), this leaks `HttpClient` selector and worker threads.
2. **ScheduledExecutor Leak**: `TransferEngine` creates new scheduled executors dynamically during execution, and `scheduleIceGraceTimer()` can create a temporary executor that is never shut down.

We will share a single static `HttpClient` instance across all `SignalingClient` instances, and refactor the scheduled executor in `TransferEngine` to be created once per transfer session and cleanly disposed.

### [TransferEngine]

#### [MODIFY] [TransferEngine.java](file:///x:/Projects_X/0_Active/1_Java_Active/AI_Assisted/WormSink_Java_GITHUB/wormsink-transfer/src/main/java/org/wormsink/transfer/TransferEngine.java)

- Initialize `heartbeatScheduler` once at the beginning of `send()` and `receive()`.
- Shut down `heartbeatScheduler` once at the end of `send()` and `receive()`.
- Modify `startSenderHeartbeat()`, `startSenderHeartbeatWatch()`, and `startReceiverHeartbeatWatch()` to reuse the pre-initialized `heartbeatScheduler` without recreating it.
- Modify `scheduleIceGraceTimer()` to directly use `heartbeatScheduler`.
