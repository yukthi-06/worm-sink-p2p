# Goal Description

Fix the deadlock/hang when restarting a receiver for a transfer that has already been 100% completed. Currently, the completion check and completion signaling (sending the `COMPLETE` frame) only occur inside the `CHUNK` handler. If the file is already fully downloaded, the receiver receives 0 chunks, never triggers the completion check, and both sides hang forever.

We will add a check in the receiver's `METADATA` handler: if the loaded transfer state shows that all chunks are completed, it will verify the integrity hash immediately and complete the session.

## Proposed Changes

### [TransferEngine]

#### [MODIFY] [TransferEngine.java](file:///x:/Projects_X/0_Active/1_Java_Active/AI_Assisted/WormSink_Java_GITHUB/wormsink-transfer/src/main/java/org/wormsink/transfer/TransferEngine.java)

- In the receiver's `handleProtocolFrame(METADATA)` case:
  - Add a check `if (transferState.completedChunks.size() == transferState.totalChunks)` right after setting `bytesTransferred`.
  - If true, calculate the file's SHA-256 hash. If it matches `transferState.fileHash`, delete the state file, send the `COMPLETE` frame to the sender, and release the latches.
  - If the hash mismatch occurs, set `sessionError` and release the latches.
