```

Download failed: Sender heartbeat timeout

Failed to attach thread 8236
```

---

## Root Cause

The receiver was dying with `"Sender heartbeat timeout"` after ~30 seconds even though the sender
was actively sending chunk data (download was progressing at 1.8 MB/s).

### Why heartbeats were not arriving

`sendChunksAsync()` runs on the ForkJoinPool common pool and monopolizes the DataChannel's
native send pipeline with large chunk frames. It uses a back-pressure spin loop
(`while bufferedAmount > 2MB: sleep(10ms)`) that holds the native send path busy.

The scheduler-based `startSenderHeartbeat()` fires every 2 s on its own daemon thread, but
calls the **same `connectionManager.sendMessage()`** path. When the ForkJoinPool thread is
continuously pushing chunks (with the back-pressure loop never fully clearing), the heartbeat
frames are effectively **starved** — they never get a send slot before the 8 s timeout fires
on the receiver.

The secondary `"VM attach current thread failed"` / `"Failed to attach thread 8236"` messages
are a known benign JNI cleanup race, unrelated to the download failure.

---

## Fix — Two-Pronged Approach

### Fix 1 — Inline heartbeat in `sendChunksAsync` (Sender side)

Inside the `sendChunksAsync` chunk-send loop, track the last time a heartbeat was sent
(`lastInlineHeartbeatMs`). Before sending each chunk, check if `HEARTBEAT_INTERVAL_MS` (2 s)
has elapsed since the last heartbeat. If yes, inject a `HEARTBEAT` frame inline into the
stream before the chunk. This guarantees heartbeats are delivered in-band with the chunks,
regardless of scheduler contention.

### Fix 2 — Reset heartbeat timer on CHUNK receipt (Receiver side)

In `handleProtocolFrame`, for the `CHUNK` case (receiver branch), immediately update
`lastHeartbeatReceivedMs = System.currentTimeMillis()` before processing the chunk.
Any arriving chunk frame proves the sender is alive. This eliminates false timeouts even
in scenarios where the explicit HEARTBEAT frame is delayed due to sender-side back-pressure.
