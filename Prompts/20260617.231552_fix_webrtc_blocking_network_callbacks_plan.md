## Goal
Fix the native JNI thread attachment failures and connection hangs by making all network calls from WebRTC callbacks asynchronous (non-blocking).

## Root Cause Analysis
- Currently, callbacks like `onIceCandidate`, and the `onSuccess` handlers for setting local descriptions, make synchronous HTTP requests (e.g. `sendCandidate`, `sendOffer`, `sendAnswer`) using Java's `HttpClient`.
- Since these callbacks are executed directly on native WebRTC threads (worker/signaling threads), the WebRTC engine's threads get blocked waiting for the network responses.
- Blocking native WebRTC threads can easily lead to deadlock, thread starvation, and JNI env attachment failures (`VM attach current thread failed`) when the engine tries to notify subsequent events.

## Proposed Changes

### [PeerConnectionManager](file:///x:/Projects_X/0_Active/1_Java_Active/AI_Assisted/WormSink_Java_GITHUB/wormsink-webrtc/src/main/java/org/wormsink/webrtc/PeerConnectionManager.java)
- Wrap the network call inside `onIceCandidate` with `CompletableFuture.runAsync()`.
- Wrap the network call inside the `onSuccess` callback of the `setLocalDescription` observer in `createOffer` with `CompletableFuture.runAsync()`.
- Wrap the network call inside the `onSuccess` callback of the `setLocalDescription` observer in `setRemoteOfferAndCreateAnswer` with `CompletableFuture.runAsync()`.
