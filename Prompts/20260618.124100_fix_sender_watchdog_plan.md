# Goal Description

Fix the deadlock/hang when the receiver is killed and restarted with the same 4-word code. The receiver gets stuck at `Connecting to peer...` because the sender does not detect the receiver's death, does not reset the signaling session, and does not start a new WebRTC offer negotiation.

## Proposed Changes

### [TransferEngine]

#### [MODIFY] [TransferEngine.java](file:///x:/Projects_X/0_Active/1_Java_Active/AI_Assisted/WormSink_Java_GITHUB/wormsink-transfer/src/main/java/org/wormsink/transfer/TransferEngine.java)

- Update `onMessageReceived` to update `lastHeartbeatReceivedMs = System.currentTimeMillis();` for both sender and receiver on any incoming data message.
- Modify `handleProtocolFrame` for `HEARTBEAT`: when the receiver gets a heartbeat from the sender, it replies with a heartbeat frame back to the sender.
- Implement `startSenderHeartbeatWatch()` on the sender:
  - Run a periodic check every 2 seconds.
  - If no messages/heartbeats are received from the receiver for 8 seconds (`HEARTBEAT_TIMEOUT_MS`), assume the receiver died.
  - Set `receiverDisconnected = true`, set `sessionError`, asynchronously call `signalingClient.resetSession()`, and count down `sessionLatch` to trigger the reconnect/re-offer loop.
- Call `startSenderHeartbeatWatch()` when the data channel opens for the sender in `onDataChannelStateChange()`.
