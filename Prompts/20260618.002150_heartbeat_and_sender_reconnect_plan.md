## Feature 1 — Heartbeat (SCTP buffer drain prevention)

### Root cause of original problem
WebRTC SCTP can buffer hundreds of MB of data on the receiver side. When the sender is
killed at 200 MB (ACK-based progress), the receiver continues downloading from its SCTP
buffer until it's exhausted (observed: sender dies at 200 MB, receiver continues to 500 MB).

### Implementation
- **Sender** (`TransferEngine`): After DataChannel OPEN, schedules `ProtocolSerializer.encodeHeartbeat()` every **2 seconds** on a daemon `ScheduledExecutorService`.
- **Receiver** (`TransferEngine`): Tracks `lastHeartbeatReceivedMs`. A watcher thread checks every 2 s; if silence > **8 seconds**, sets `sessionError = "Sender heartbeat timeout"` and counts down `sessionLatch` immediately — stopping the download without draining the SCTP buffer.
- `HEARTBEAT` frame type (byte `9`) already existed in `ProtocolMessage` and `ProtocolSerializer`.

## Bug 2 — Receiver restart stuck at "Connecting to peer"

### Root cause
When receiver is killed, the signaling server still holds the **old offer + answer + ICE candidates** from the first connection attempt. When the new receiver polls, it receives the old offer and creates a new answer. But the sender:
1. Already set the first receiver's answer as the remote description → WebRTC state machine won't accept a second answer.
2. Stopped polling `pollForAnswer()` after the first receiver connected.
3. No longer processes ICE candidates from the new receiver.

Result: new receiver is stuck waiting for a data channel that the sender will never open.

### Implementation — two parts

#### Part A — Signaling server (`signaling-server-cloudflare/index.js`)
Added `POST /session/reset` endpoint that clears `offer`, `answer`, and `candidates` while keeping the session alive. This lets the sender post a fresh offer that a new receiver can pick up.

#### Part B — Sender auto-reconnect (`TransferEngine.java`)
Replaced the single `CountDownLatch completeLatch` (can't be reset) with a **per-attempt** `volatile CountDownLatch sessionLatch` that is replaced on each reconnect.

Added a reconnect loop in `send()`:
1. On ICE/connection `FAILED` → `receiverDisconnected = true; sessionLatch.countDown()`
2. The `send()` loop detects `receiverDisconnected == true`, calls `SignalingClient.resetSession(code)`, sleeps 1 s, then creates a fresh `PeerConnectionManager` (new offer → new WebRTC negotiation).
3. The receiver, on next start with the same code, polls and finds the fresh offer → connects normally.

#### Part C — `SignalingClient.java`
Added `resetSession(String code)` method that POSTs to `/session/reset`.
