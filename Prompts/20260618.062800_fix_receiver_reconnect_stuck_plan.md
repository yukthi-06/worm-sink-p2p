# Fix: Receiver Stuck on Reconnect (Stale Offer / ICE Limbo)

## Root Cause Analysis

The three output lines printed are **all pre-connection** messages — they fire immediately at the start of `handleReceive()` / `receive()` before any WebRTC activity. The receiver is **blocking forever on `sessionLatch.await()`** — the latch is never counted down.

### Why the latch is never counted down

The latch is released by:
1. `onConnectionStateChange()` — only if state contains `"FAILED"` or `"CLOSED"`
2. `onDataChannelStateChange(CLOSED)` — only if data channel was ever opened
3. Heartbeat timeout — only starts **after** data channel OPEN
4. Protocol error

**If ICE gets stuck in `DISCONNECTED` and never transitions to `FAILED`, none of the above fire.**

### The race condition that causes ICE limbo

```
T+0     Sender is streaming chunks. Receiver is killed.
T+0     New receiver process starts immediately.
T+0     Receiver polls /session/{code} → finds the OLD offer (still present).
T+0     Receiver processes old offer, posts an answer, starts ICE.
T+5s    Sender's ICE detects failure (ICE DISCONNECTED → FAILED).
T+5s    Sender: stopHeartbeat() + connectionManager.close() → takes ~3.4s.
T+8.4s  Sender: resetSession() → WIPES the new receiver's answer!
T+9.4s  Sender: sleep(1s) → new PeerConnectionManager → new offer posted.
```

At this point the receiver is trying to negotiate ICE with credentials from the **old session**. Since the sender (still alive) partially responds to STUN probes, the receiver's ICE state goes `DISCONNECTED` but **never reaches `FAILED`** — it oscillates, and the receiver hangs forever.

## Proposed Fixes (Three Layers)

### Fix 1 — Early Signaling Reset (Root Cause Prevention)
**File:** `TransferEngine.java` — sender reconnect loop

Move `resetSession()` to happen **immediately** when the sender detects receiver disconnect, **before** `connectionManager.close()`. This ensures the stale offer is gone before any new receiver process can poll.

```
Before: detect → close(3.4s) → resetSession → sleep(1s) → new offer
After:  detect → resetSession (async fire-and-forget) → close(3.4s) → sleep(1s) → new offer
```

### Fix 2 — Receiver Connection Timeout
**File:** `TransferEngine.java` — `receive()` method

Replace the blocking `sessionLatch.await()` with `sessionLatch.await(60, TimeUnit.SECONDS)`.
If the timeout fires, set `sessionError` to `"Connection timed out (60s)"` and allow the engine to exit cleanly.

### Fix 3 — ICE DISCONNECTED Grace Timeout
**File:** `TransferEngine.java` — `onConnectionStateChange()`

If the receiver sees `ICE: DISCONNECTED` while the data channel is not yet open, schedule a one-shot 15s timer. If no improvement (OPEN / FAILED / CLOSED) occurs in that window, force-count-down the latch.
