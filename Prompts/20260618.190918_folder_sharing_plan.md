## Initial Analysis
We want to support sharing a full directory/folder across the peer-to-peer network without compressing it into a temporary archive (which is slow and inefficient for GB-scale data).
To achieve this, we will implement **Virtual File Concatenation**:
1. Treat the entire directory structure as a single virtual contiguous file of size $N$ (the sum of all file sizes).
2. Scan the directory recursively on the sender side, sorting files alphabetically by their relative path to establish a deterministic ordering.
3. Build a manifest mapping relative file paths to their virtual offset ranges.
4. Pre-allocate all folders and empty files on the receiver side upon receiving the manifest.
5. Translate all read and write requests dynamically from virtual offsets to the physical files and offsets.
6. Support resumes and block hashes over the virtual file exactly like we do for single files.

## Assumptions
- The files inside the directory do not change in size or content during the lifetime of the sharing session.
- The relative paths use forward slashes for cross-platform portability.

## Risks
- **Very large directories (millions of files)**: The JSON manifest payload could grow large. However, for normal folder shares containing thousands of files, this will fit well within signaling payloads and standard memory limits.
- **Empty files / Zero-length files**: Our virtual mapping code must handle files of zero size or chunks spanning across file boundaries cleanly.
