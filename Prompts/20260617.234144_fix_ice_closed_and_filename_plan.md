## Issue 1 — "ICE: CLOSED" false error after successful transfer

### Root cause
After `completeLatch.countDown()` fires on the success path, `send()/receive()` calls
`connectionManager.close()`. This teardown causes the native WebRTC engine to asynchronously
fire `onIceConnectionChange(CLOSED)` → `onConnectionStateChange("ICE: CLOSED")`. Since
`errorMessage` was null (transfer succeeded), the handler sets it to `"ICE: CLOSED"` and
the transfer reports failure instead of success.

### Fix — TransferEngine.java
- Add `private volatile boolean transferCompleted = false;`
- Set `transferCompleted = true` **before** `completeLatch.countDown()` on all success paths
  (receiver file-complete, sender COMPLETE receipt).
- Add `if (transferCompleted) return;` at the top of `onConnectionStateChange`.
- Guard the DataChannel CLOSED error message with `!transferCompleted`.

## Issue 2 — Downloaded file named after the transfer code

### Root cause
`WormSinkCli.handleReceive()` defaults `outputPath = "./" + code + ".download"`. This
placeholder is passed as `destinationPath` to `TransferEngine`, which never updates it to
use the actual filename from METADATA.

### Fix
- **WormSinkCli.java**: Default `outputPath = "."` (current directory).
- **TransferEngine.handleProtocolFrame(METADATA)**: When `destinationPath` is a directory,
  resolve to `new File(destinationPath, fileName).getPath()` and update `stateFilePath`
  accordingly.
- **WormSinkCli.java onTransferStarted**: Print "Saving to: `<dir>/<name>`" by computing
  the resolved path the same way TransferEngine does.
