## Root Cause Identified

The signaling server returns JSON like this for a newly-created session:
```json
{"code":"...", "offer":null, "answer":null, "candidates":[]}
```

`SimpleJson.getField()` uses the regex `([^,}\s]+)` to match unquoted values. This matches
the literal text `null` in JSON and returns the Java **String** `"null"` — NOT Java `null`.

As a result:
- `session.offer != null` is **always true** even before the offer is sent.
- `session.answer != null` is **always true** even before the answer is sent.

Both `pollForOffer` and `pollForAnswer` fire immediately on their first iteration (0-delay
`scheduleWithFixedDelay`) and call `peerConnection.setRemoteDescription("null")` with an
invalid SDP. The native WebRTC engine hangs indefinitely on the invalid SDP, never calling
onSuccess or onFailure — causing the permanent hang both users see.

## Proposed Fix

### [SimpleJson.java](file:///x:/Projects_X/0_Active/1_Java_Active/AI_Assisted/WormSink_Java_GITHUB/wormsink-signaling/src/main/java/org/wormsink/signaling/SimpleJson.java)

In `getField()`, after resolving `val`, add a guard:
```java
if ("null".equals(val)) {
    return null;
}
```
This correctly maps the JSON null literal to Java null.
